## 🚨 ПРАВИЛО НОЛЬ — НЕ ЧИТАЙ РЕЗУЛЬТАТЫ АГЕНТОВ В СВОЙ КОНТЕКСТ

**Crabe (оркестратор) НИКОГДА не делает `Read` на файлы результатов агентов.**

Запрещено читать в основной контекст:
- ❌ Протоколы жюри (`scores/jury-*.md`)
- ❌ Сырые идеи креативщиков (`raw/creator-*.md`)
- ❌ Отчёты исследователей (`research/*.md`)
- ❌ Заключения аналитиков (`analysis/analyst-*.md`)
- ❌ Логи агентов (`logs/agent-*.log`)
- ❌ Любые файлы >2KB из `results/`

**Что делать вместо этого:**
- Спавнить суб-агента для обработки → получать только краткую сводку
- Проверять существование файла через `ls -la` (размер > 0 = ок)
- Единственное исключение: `scored.md` и `executive-summary.md` — итоговые файлы, созданные специально для Crabe

**Нарушение = мгновенное переполнение контекста = Сергей чистит за тобой руками.**

## Главный принцип: "БД — это память агентов, файлы — для людей"

**Агенты НЕ общаются через чат.** Каждый агент:
1. Получает задание (только нужный контекст, ~5-15K токенов)
2. **Читает из БД** (если нужны данные от предыдущего этапа) — не файлы
3. **Пишет чекпойнты в БД** по мере работы (не файл в конце)
4. Умирает — частичная работа уже сохранена в БД

**Оркестратор (Crabe) НЕ читает результаты агентов в свой контекст.**
Вместо этого: `SELECT COUNT(*) FROM researcher_checkpoints WHERE status='success'` → запускает следующего.

### Экономия
- v1: Opus, общий контекст ~300K, стоимость ~$31
- v2: Sonnet, изолированный контекст ~5-15K, стоимость ~$2-4
- v3: Sonnet + DB, контекст аналитиков ~10-20KB вместо ~320KB, краши — recoverable

## ⛔ Жёсткие ограничения (фильтры-отсекатели)

Идея **автоматически выбывает** если нарушает хотя бы одно:

1. **Без большого рекламного бюджета** — рост только через органику: SEO, word-of-mouth, community, Product Hunt, холодный outreach. Нет денег на масштабную платную рекламу.
2. **Простой код** — MVP собирается через вайбкодинг (Claude/Cursor + no-code: n8n, Bubble, Webflow, Framer). Один человек без опыта программирования за 2-4 недели.
3. **Без специальной экспертизы** — не требует профессиональных знаний в закрытых областях: медицина, юриспруденция (практика), фармацевтика, финансовое консультирование с лицензией.
4. **Без лицензий** — запуск немедленно. Никаких разрешений и сертификаций которые занимают месяцы.

Эти фильтры применяются на **этапе 3 (оценка жюри)** — перед выставлением баллов.

## Как запускать агентов (шаблон)

```
sessions_spawn(
  mode = "run",                          # одноразовая сессия
  model = "anthropic/claude-sonnet-4-6", # дешёвая модель
  task = """
    [Роль и характер агента — копируем из config/]
    
    ЗАДАНИЕ: [конкретное задание]
    
    ВХОДНЫЕ ДАННЫЕ: Прочитай файл {путь} если нужен контекст от предыдущего этапа.
    
    РЕЗУЛЬТАТ: Запиши ВЕСЬ результат в файл:
    {путь к выходному файлу}
    
    ФОРМАТ: [описание формата]
    
    Когда закончишь — просто заверши работу. Ничего не возвращай в чат.
  """
)
```

### Критически важно:
- **task** содержит ВСЁ что нужно агенту — роль, задание, пути к файлам
- Агент **читает** входные файлы сам (через Read)
- Агент **пишет** результат в файл (через Write)
- Оркестратор **НЕ читает** результат в свой чат
- Проверка: `ls -la {путь к файлу}` — файл есть и размер > 0

### ⚠️ Баг с именами в логах — исправление
**ПЕРЕД** вызовом sessions_spawn для каждого агента — записать первую строку в его лог-файл с ПРАВИЛЬНЫМ именем:
```
echo "[$(date '+%Y-%m-%d %H:%M:%S')] 🚀 {ИМЯ_АГЕНТА} запущен — {ЗАДАНИЕ}" >> {LOG_FILE}
```
Так имя в дашборде всегда будет правильным, даже если агент напишет в лог с задержкой.

## Pipeline: порядок выполнения

### Фаза 1: Генерация и отбор

**ЭТАП 1: Генерация идей**
- Запуск: 12 креативщиков параллельно (`sessions_spawn` × 12)
- Каждый получает: свой профиль из config + задание + путь для записи
- Каждый генерирует 20 идей
- Результат: `{run}/01-ideas/raw/creator-XX-name-ideas.md`
- Проверка: 12 файлов появились, размер > 1KB
- **Ждём ВСЕХ**

**ЭТАП 2: Дедупликация**
- Перед спавном: `echo "[timestamp] 🚀 Дедупликатор запущен — обрабатываю {N} идей" >> {run}/logs/agent-ops-01.log`
- Запуск: 1 агент (Дедупликатор), промпт из `prompts/operations/deduplicator.md`
- Агент в конце пишет в лог: `[timestamp] ✅ Дедупликатор завершил [FILE:{run}/01-ideas/deduplicated.md]`
- Проверка: файл есть, содержит ID-нумерацию
- **Ждём завершения**

**ЭТАП 3: Оценка жюри**
- Запуск: 10 членов жюри параллельно
- Каждый получает: свой профиль + "Прочитай `{run}/01-ideas/deduplicated.md`. Оцени каждую идею по 8 критериям (1-10). Запиши в `{run}/01-ideas/scores/jury-XX-name.md`"
- Формат оценки: `ID-001: [8,7,6,9,5,8,7,6] — комментарий`
- Проверка: 10 файлов оценок
- **Ждём ВСЕХ**

**ЭТАП 4: Подсчёт и рейтинг**
- Перед спавном: `echo "[timestamp] 🚀 Считовод запущен — подсчёт оценок от {N} судей" >> {run}/logs/agent-ops-02.log`
- Запуск: 1 агент (Считовод), промпт из `prompts/operations/scorecounter.md`
- Агент в конце пишет в лог: `[timestamp] ✅ Считовод завершил [FILE:{run}/01-ideas/scored.md]`

> Архивариус удалён из pipeline (v2: каждый агент пишет файл сам).
- **Ждём завершения**

**ЭТАП 5: Отбор топ-20**
- Выполняет: Crabe (я)
- Читаю ТОЛЬКО `scored.md` (единственный файл который я открываю в контекст)
- Показываю Сергею топ-20
- После утверждения → создаю `{run}/02-top20/top20-list.md` + папки `idea-XX/brief.md`

**⏸️ ПАУЗА: Сергей утверждает топ-20**

### Фаза 2: Исследование и анализ (потоково, 2-3 идеи одновременно)

**ЭТАП 6: Исследование** _(v3: чекпойнт-модель)_
- По каждой идее: 16 исследователей параллельно
- task: "Прочитай `{run}/02-top20/ideas/idea-XX/brief.md`. [Твоя роль]. Найди информацию.  
  **ВАЖНО:** Пиши каждую находку в БД через `05-research.js:saveCheckpoint()`, не жди конца работы."
- Исследователь делает:
  1. `initCheckpoint()` → INSERT {status: in_progress}
  2. Per finding: `saveCheckpoint(text, confidence, sources)` → UPDATE
  3. `completeCheckpoint()` → UPDATE {status: success}
  4. Таймаут? → статус остаётся `in_progress`, watchdog переведёт в `timeout`
- Проверка оркестратора: `SELECT status, COUNT(*) FROM researcher_checkpoints WHERE idea_id=$1 GROUP BY status`
- **Ждём всех 16 по данной идее** (или таймаут + пометка)

**ЭТАП 6.5: Обновление coverage_matrix**
- Оркестратор вызывает `05-research.js:updateCoverageMatrix(ideaId)` после финиша исследователей
- coverage_matrix показывает какие области покрыты: market_analysis / user_research / competitive / financial / regional
- Если критические области `✗` — можно перезапустить конкретных исследователей, не всех

**ЭТАП 7: Чистка** _(упрощена в v3)_
- 1 агент (Чистильщик) на идею
- task: "Прочитай `researcher_checkpoints WHERE idea_id=$1 AND echo_depth=3`.  
  Это записи где 3+ исследователей сообщили похожее — проверь источники, это context echo или реальные данные?  
  Обнови confidence в БД если находки были мусором."
- Физическая чистка файлов больше не нужна — данные уже структурированы в БД

**ЭТАП 8: Аналитика** _(v3: запросы к БД вместо чтения файлов)_
- 10 аналитиков параллельно на идею
- task: "Используй `06-analysis.js:buildAnalystContext(ideaId)` — получишь структурированный контекст из БД.  
  Запрос автоматически фильтрует: confidence >= 0.6, echo_depth ASC, confidence DESC.  
  [Твоя роль]. Напиши заключение на русском. Запиши в `{run}/02-top20/ideas/idea-XX/analysis/analyst-XX-name.md`"
- Контекст аналитика: ~10-20KB (против ~320KB раньше)

**ЭТАП 9: Консолидация**
- 1 агент (Верховный аналитик) на идею
- task: "Прочитай ВСЕ файлы из `{run}/02-top20/ideas/idea-XX/analysis/`. Напиши summary.md (1-2 абзаца, 🟢/🟡/🔴) и consolidated.md (развёрнутый отчёт). Запиши в `{run}/02-top20/ideas/idea-XX/`"

### Фаза 3: Финализация

**ЭТАП 10: Сборка**
- 1 агент (Верховный аналитик)
- task: "Прочитай ВСЕ summary.md из `{run}/02-top20/ideas/*/`. Собери executive-summary.md + full-report.md + dashboard.html. Запиши в `{run}/03-final/`"

**ЭТАП 11: Доклад Сергею**
- Crabe читает ТОЛЬКО executive-summary.md и докладывает

## 💾 Persistent State — Система чекпойнтов (v3)

> Зачем: исследователи работают 15-60 минут. Любой краш = потеря работы.  
> v3 решение: каждая находка сохраняется в БД **немедленно**, не ждёт конца.

### Жизненный цикл чекпойнта

```
[researcher starts]
    ↓
INSERT researcher_checkpoints {status: in_progress}
    ↓
[finding #1 discovered]
    ↓
UPDATE {result_text, confidence, evidence_sources, echo_depth}   ← живая запись
    ↓
[finding #2...]
    ↓
UPDATE {result_text...}   ← перезаписывает, всегда последнее
    ↓
[researcher completes normally]
    ↓
UPDATE {status: success}
    
--- ЕСЛИ КРАШ ---
status остаётся 'partial' → данные НЕ потеряны
Watchdog видит 'in_progress' > 60 минут → UPDATE {status: timeout}
Оркестратор перезапускает → ON CONFLICT DO UPDATE → продолжает с нуля (или backup agent читает partial)
```

### echo_depth — обнаружение context echo

```
echo_depth = 1   → уникальная находка (первый кто нашёл)
echo_depth = 2   → подтверждённая (1-2 других тоже нашли) → выше доверие
echo_depth = 3   → эхо (3+ нашли похожее) → ПРОВЕРИТЬ ИСТОЧНИКИ
                   Проблема: все могли прочитать один AI-сгенерированный текст
```

**Правило:** echo_depth=3 findings идут в `06-analysis.js:fetchEchoedFindings()` для дополнительной проверки.

### Состояние которое оркестратор НЕ хранит у себя
- Прогресс исследователей → `researcher_checkpoints`
- Покрытие по областям → `coverage_matrix`
- Оценки жюри → `jury_evaluations`
- Варианты идей → `idea_variants`

Оркестратор только **читает статусы** через SQL, **не держит в памяти**.

## ⚖️ Jury Weighting — Модель автономии (v3)

> Прежде: все 10 судей оценивали по одинаковым 9 критериям с равными весами.  
> v3: каждый судья имеет собственную функцию ценности.

### Почему это лучше

Стартапер и Финансист смотрят на разные вещи. Раньше оба ставили "scalability" с одинаковым весом — это неправда. Теперь:
- Стартапер: launch_difficulty × 0.40 (его главный вопрос)
- Финансист: profitability × 0.40 (его главный вопрос)

Итоговый score = взвешенное среднее по матрице `jury_criteria_weights`.

### Три фазы калибровки

```
Phase 1 (идеи 1-20):   Все голоса равны. Нет весов. Принимаем шум.
Phase 2 (идеи 21-40):  Ретроактивная проверка точности. "Jury-A точен на 80% по scalability"
Phase 3 (идеи 41+):    Взвешивание по Phase 2 калибровке.
```

> Доверие покупается наблюдениями, не предположениями. (cold start trust model)

### Высокое расхождение = сигнал, не баг

```sql
-- Найти поляризующие критерии для идеи
SELECT criterion_name,
       ROUND(AVG(score)::numeric, 2)    AS avg,
       ROUND(STDDEV(score)::numeric, 2) AS stddev
FROM jury_evaluations
WHERE idea_id = 'idea-03'
GROUP BY criterion_name
HAVING STDDEV(score) > 2.0
ORDER BY stddev DESC;
```

Если один судья ставит 9/10, другой 3/10 на scalability:
- **Не** фильтровать и усреднять
- **Да** — исследовать почему. Возможно idea-03 нужно разбить на `idea_variants` (B2B vs B2C версии)

Подробнее — `config/jury-team-v3.md`.

## 🫀 Heartbeat Monitoring — Мониторинг таймаутов (v3)

> v2: проверяли "файл появился?". v3: запрашиваем БД.

### Оркестратор проверяет каждые ~10 минут

```javascript
// Проверка прогресса исследования
const { rows } = await pool.query(`
  SELECT status, COUNT(*) 
  FROM researcher_checkpoints 
  WHERE idea_id = $1 
  GROUP BY status
`, [ideaId]);
// → { in_progress: 3, partial: 8, success: 5, timeout: 0 }
```

### Watchdog для stuck agents

```javascript
// Найти завязших (in_progress > 60 минут)
await detectTimeouts(60);  // из 05-research.js
// Автоматически UPDATE status='timeout' для старых in_progress
```

### Решения при таймауте исследователя

| Ситуация | Действие |
|---|---|
| timeout + result_text IS NOT NULL | Partial работа есть — аналитики используют её |
| timeout + result_text IS NULL | Ничего нет — перезапустить исследователя |
| echo_depth=3 на этом researcher | Backup researcher уже нашёл то же самое — можно не перезапускать |
| Все 5 исследователей одной области = timeout | Критично — обязательный перезапуск |

### coverage_matrix как сводка для оркестратора

```sql
SELECT * FROM coverage_matrix WHERE idea_id = 'idea-03';
-- idea-03 | market_analysis: ✓ | user_research: partial | competitive: ✓ | financial: ✗ | regional: ✓
```

Если `financial_analysis = ✗` — перезапустить финансового исследователя перед аналитикой.
